Investigation T-042 finding: review runs sandboxed by default (review.rs acquire_sandbox), but the heyvm sandbox is a headless ubuntu:24.04 microVM with only filesystem mounts and NO Wayland/DISPLAY forwarding (plugins/heyvm/printer-plugin.toml). Inside it $WAYLAND_DISPLAY/$XDG_SESSION_TYPE are unset and /dev/uinput is absent, so the computer CLI's Connection::connect_to_env() (computer/src/platform/linux) cannot drive anything and the review agent (correctly) reports 'no display' and skips click-testing. Fix: detect when a diff touches a UI/web surface and run the review phase on the host (review.rs already supports --no-sandbox) where a Wayland session can exist, OR document the --no-sandbox UI-review workflow in the review prompt/README. Acceptance: a UI change can be click-tested during review without manual flag-twiddling, or the manual workflow is documented.

Goal: auto-route the review phase to the HOST (no sandbox) when the diff touches a UI/web surface AND the host has a real display, so the computer tool can click-test without manual --no-sandbox.

New helpers in printer/src/review.rs:
- fn ui_surface_changed(cwd: Option<&Path>, base: &str) -> bool: run 'git diff --name-only {base}...HEAD' + 'git status --porcelain' (untracked/unstaged) in cwd; true if any path matches a conservative UI allowlist: exts .tsx/.jsx/.vue/.svelte/.html/.htm/.css/.scss/.sass + (.ts/.js only under web/ frontend/ ui/ client/ src/components/ app/ or beside an index.html). Keep matcher a pure fn over Vec<String> so it is unit-testable without git.
- fn host_display_available() -> bool: mirror computer/src/platform/linux Connection::connect_to_env preconditions — true iff (WAYLAND_DISPLAY set OR XDG_SESSION_TYPE in {wayland,x11}) AND Path::new('/dev/uinput').exists().

Wire into acquire_sandbox(args): after the no_sandbox/driver-resolve guards, if ui_surface_changed(cwd,base) && host_display_available() -> log '[printer] UI/web surface detected + host display present; running review on host (no sandbox) for click-testing' and return Ok(None). acquire_sandbox currently doesn't compute base — thread base in (args.base or detect_base) so the diff range matches the reviewer's.
Escape hatch: add CLI flag --no-ui-host (ReviewArgs in cli.rs, default false) to force sandbox even for UI diffs; honor it in the branch.
Note: 'printer exec' shares one sandbox across run+review via review_with_sandbox(sandbox), bypassing acquire_sandbox — host-routing only applies to standalone 'printer review'. Document that exec UI review needs --no-sandbox. Add a README review-section sentence covering the auto-host behavior + --no-sandbox manual fallback (satisfies the 'or document' half of acceptance).
